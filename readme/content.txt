  По DONE для Motion текущая картина такая:

  - DONE уже корректен для сервисных команд, STOP, SET_SPEED, SET_ACCELERATION.
  - DONE не закрыт для finite-команд ROTATE и настоящего HOME.
  - Причина не в CAN-логике, а в движении: сейчас ROTATE запускает PWM, но firmware не считает фактически выданные STEP-импульсы, не останавливает канал по достижению steps, и поэтому не имеет корректной точки для отправки DONE.

  Правильное решение по архитектуре:

  1. ROTATE
      - Dispatcher принимает Host/Conductor команду.
      - MotionController проверяет motor busy, задает направление, число шагов, скорость.
      - Driver запускает PWM/STEP.
      - Каждый STEP считается из timer interrupt / PWM compare callback.
      - Когда steps_done == target_steps, PWM останавливается, EN переводится в безопасное/idle состояние, позиция обновляется.
      - MotionController отправляет DONE.
  2. START_CONTINUOUS
      - Это не finite-команда.
      - DONE может означать “continuous mode включен”.
      - Завершение режима идет отдельной командой STOP.
      - STOP отправляет DONE после фактической остановки.
  3. HOME
      - Нельзя корректно закрыть как DONE, пока нет источника home-condition.
      - Нужен концевик/датчик/индекс/другая физическая точка.
      - Без датчика можно сделать только “виртуальный home”, но это не настоящий HOME.

  Главный инженерный выбор перед реализацией:

  - Вариант A, быстрый no-load baseline: считать STEP через PWM interrupt, остановить после N импульсов, отправить DONE. Подходит для проверки протокола и базового finite move.
  - Вариант B, промышленный: добавить полноценный motion completion layer: счетчики шагов, состояние оси, события completion, future acceleration/deceleration, HOME sensor handling, fault/STOP arbitration.

  Я бы выбрал так: сначала ROTATE finite DONE как baseline через счетчик PWM events, затем отдельным этапом HOME с датчиком. Это минимально двигает проект вперед и не смешивает две разные задачи.

› ну давай тогда отделяй мух от котлет, это абсолютно два рззных случая


• Да. Нужно жестко разделить два типа команд.

  1. Finite-команды

  Это команды с внутренним естественным окончанием.

  Примеры:

  - MOTOR_ROTATE(steps, speed);
  - MOTOR_HOME(...);
  - будущие дозирующие команды типа PUMP_RUN_DURATION(duration_ms).

  Жизненный цикл:

  COMMAND
  → ACK
  → исполнитель сам выполняет физику до условия завершения
  → исполнитель сам останавливает выходы
  → DONE

  Для ROTATE условие завершения:

  remaining_steps == 0

  STOP в штатном сценарии не нужен.

  STOP для finite-команды может быть только:

  - аварийная отмена;
  - ручное вмешательство;
  - timeout/recovery;
  - пользовательская остановка до завершения.

  Если finite-команду остановили через STOP, то это уже не “нормальный DONE исходной ROTATE”. Это отдельная команда STOP со своим DONE, а исходная ROTATE должна считаться прерванной/не завершенной на уровне Conductor.

  2. State / continuous-команды

  Это команды, которые переводят устройство в состояние и не имеют собственного физического конца.

  Примеры:

  - MOTOR_START_CONTINUOUS(speed);
  - ручной PUMP_START;
  - включить нагрев/термостатирование в удержание;
  - включить клапан, если это state-команда.

  Жизненный цикл:

  COMMAND
  → ACK
  → состояние включено
  → DONE

  Здесь DONE означает не “физический процесс закончился”, а “команда смены состояния выполнена”.

  Остановка/выход из состояния отдельной командой:

  STOP / OFF / DISABLE
  → ACK
  → состояние выключено
  → DONE

  Для Motion получается так

  MOTOR_ROTATE:

  ACK → выдать N STEP → остановить PWM → обновить позицию → DONE

  MOTOR_HOME:

  ACK → двигаться до home-condition → остановить PWM → обновить home/position → DONE

  MOTOR_START_CONTINUOUS:

  ACK → включить continuous PWM → DONE

  MOTOR_STOP:

  ACK → остановить active/continuous/finite motion → DONE

  Главное правило

  DONE всегда означает: данная команда выполнила свой контракт.

  Но контракт у команд разный:

  - finite-команда: завершилась физика;
  - state-команда: состояние успешно установлено;
  - stop-команда: состояние/движение успешно прекращено.

  В отчет добавлен отдельный раздел про разделение:

  - finite-команды: ROTATE, HOME, PUMP_RUN_DURATION завершаются сами по своему физическому условию и только потом дают DONE;
  - state/continuous-команды: START_CONTINUOUS, PUMP_START, STOP/OFF дают DONE после смены состояния;
  - STOP не является штатным завершением ROTATE.

  План на завтра в readme/refactoring_plan.md:533 тоже уточнил: следующий практический шаг — MOTOR_ROTATE completion через счетчик STEP-событий текущего TIM1/TIM2 PWM path, с DONE из task context, не из ISR.

  Код после твоего замечания не трогал. На сегодня точка остановки понятная: safety закрыта без нагрузки, дальше не ROTATE + STOP, а самостоятельный finite ROTATE -> DONE.
